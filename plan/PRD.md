# PRD: Mafia Pick'em

## 1. Обзор продукта

**Название:** Mafia Pick'em  
**Платформа:** Telegram Mini App (mobile-first, «второй экран»)  
**Цель:** Повысить вовлеченность зрителей прямых трансляций по спортивной мафии через интерактивные прогнозы с соревновательным рейтингом.  
**Концепция:** Зрители соревнуются в чтении игры, делая прогнозы на «нулевом круге» с открытыми ролями. Модуль работает по принципу фэнтези-компендиума (Pick'em), где пользователи накапливают баллы за успешные прогнозы и поднимаются в таблице лидеров турнира.

## 2. Бизнес-модель

Доступ к интерактиву **бесплатный** для всех зарегистрированных пользователей. Единственное условие участия — авторизация через Telegram и указание игрового никнейма.

Баллы не имеют финансовой ценности и не могут быть конвертированы в деньги. Интерактив создан исключительно в развлекательных целях (For entertainment purposes only).

По итогам турнира Топ-X зрителей из Leaderboard могут получать физические призы (мерч, атрибутика) по усмотрению организаторов. Денежные выплаты запрещены.

## 3. Пользовательские роли

| Роль | Описание |
|------|----------|
| **Registered User** | Прошел Telegram-авторизацию и указал `GameNickname`. Делает прогнозы, участвует в рейтинге. |
| **Admin** | `TelegramId` входит в конфигурационный allowlist (`PickemAdminTelegramIds`). Управляет турнирами, матчами, переключает статусы, фиксирует результаты. Функции админа встроены прямо в интерфейс Mini App (не отдельная панель). |

## 4. Авторизация и регистрация

### 4.1. Аутентификация

1. Mini App запускается по прямой ссылке `t.me/...` (Web App link).
2. Telegram WebApp SDK передает `initData` с подписью.
3. Backend валидирует HMAC-SHA256 подпись `initData` с использованием Bot Token.
4. При успешной валидации выполняется upsert пользователя (по `TelegramId`) и выдается JWT-токен (HS256, срок жизни — 48 часов).
5. JWT содержит claims: `UserId`, `telegram_id`, `is_admin`, `nickname`.
6. Авторизация обязательна на старте приложения: до успешного получения JWT доступ к игровым и информационным API не предоставляется.

### 4.2. Регистрация (GameNickname)

1. При первой авторизации пользователю задается временный placeholder-никнейм (`_unregistered_{telegramId}`).
2. До сохранения `GameNickname` пользователь считается незарегистрированным и **не может отправлять прогнозы**.
3. Отдельный экран регистрации предлагает ввести `GameNickname`.
4. Правила валидации:
   - Длина: 2–30 символов.
   - Допустимые символы: `[a-zA-Z0-9 _-]` (латиница, цифры, пробел, дефис, подчеркивание).
   - Уникальность: никнейм не должен совпадать с существующим (проверяется серверно).
5. После сохранения никнейма JWT перевыпускается с обновленными claims.

### 4.3. Dev-режим

В DEBUG-сборке доступен endpoint `POST /api/auth/dev`, создающий тестового пользователя (TelegramId=999999999) для локальной разработки без Telegram.

## 5. Игровая механика и скоринг

### 5.1. Вопросы для прогноза

В каждой игре турнира участнику предлагается ответить на 2 вопроса до начала активной фазы:

1. **Кто победит в игре?** — Варианты: Мирные (Town, `0`) / Мафия (Mafia, `1`).
2. **Кого заголосуют первым?** — Варианты: Игрок 1..10 (`1`–`10`) / Никто (`0`).

Прогноз фиксируется атомарно (оба ответа в одной записи). Можно менять неограниченно, пока матч в статусе `Open`. Также можно удалить свой прогноз, пока матч `Open`.

### 5.2. Динамическая система очков

Используется тотализаторная модель: чем меньше людей угадало правильный ответ, тем больше баллов они получают. Очки рассчитываются при переходе матча в `Resolved`.

**Формулы:**

```
WinnerPoints  = (TotalVotes / CorrectWinnerVotes) × 10
VotedOutPoints = (TotalVotes / CorrectVotedOutVotes) × 20
TotalPoints   = WinnerPoints + VotedOutPoints
```

- `TotalVotes` — общее число прогнозов на матч.
- `CorrectWinnerVotes` — число угадавших победителя.
- `CorrectVotedOutVotes` — число угадавших уход.
- Базовые множители: **10** (победитель), **20** (уход — угадать сложнее).
- Защита от деления на ноль: если `CorrectVotes = 0`, соответствующие очки равны `0` (никто не угадал → никто не получает баллы).

### 5.3. Множественный уход

Если на первом голосовании стол покидают несколько игроков, модератор выбирает **несколько правильных слотов**. Правильные слоты хранятся в CSV-формате (например `"3,7"`). Баллы начисляются каждому, кто угадал **любой** из правильных слотов.

## 6. Жизненный цикл матча (стейт-машина)

### 6.1. Статусы

| Статус | Код | Описание |
|--------|-----|----------|
| **Upcoming** | `0` | Матч создан. Карточка видна, прогнозы недоступны. |
| **Open** | `1` | Прием прогнозов открыт. Отображается crowd-статистика в реальном времени. |
| **Locked** | `2` | Прогнозы заблокированы. Отображается финальная crowd-статистика. |
| **Resolved** | `3` | Результаты зафиксированы, очки начислены, лидерборд обновлен. |

### 6.2. Допустимые переходы

```
Upcoming ──→ Open           (открыть прием прогнозов)
Open     ──→ Upcoming       (откатить к ожиданию)
Open     ──→ Locked         (заблокировать прогнозы)
Locked   ──→ Open           (переоткрыть прием прогнозов)
Locked   ──→ Resolved       (зафиксировать результаты и рассчитать очки)
Resolved ──→ Locked         (откатить результаты: удалить очки, вернуть в Locked)
```

Переходы назад позволяют модератору исправить ошибки без потери данных. При переходе `Resolved → Locked` происходит rollback скоринга: удаляются `PredictionScore` и `MatchResult`, лидерборд пересчитывается.

Каждый переход записывает соответствующий timestamp (`DateOpened`, `DateLocked`, `DateResolved`).

### 6.3. Удаление матча

Модератор может удалить матч. При удалении каскадно очищаются: `PredictionScore` → `MatchResult` → `Prediction` → `Match` → blob-файл состояния.

```
    │   │   └── MatchStateControls (admin, inline)

    └── Вкладка «Лидеры»

```

### 7.2. Экран матча — состояние Open (нет прогноза)

- Плашка состояния 🟢, номер игры.
- **Блок «Кто победит?»**: две кнопки — «Мирные» / «Мафия».
- **Блок «Кого заголосуют первым?»**: сетка из 10 пронумерованных кнопок (слоты 1–10) + кнопка «Никто» (слот 0).
- **Кнопка «Сохранить прогноз»**: активна только при выборе обоих ответов.

### 7.3. Экран матча — состояние Open (прогноз сохранен)

- Отображается **crowd-статистика** (CrowdStats) с процентами голосов.
- Кнопка **«Отменить прогноз»** — удаляет прогноз и возвращает к форме.

### 7.4. Экран матча — состояние Locked

- Плашка 🔒 «Прогнозы закрыты».
- Отображается crowd-статистика:
  - Горизонтальная полоса: % Мирные / % Мафия.
  - Вертикальный столбчатый график: распределение голосов по слотам.
- Прогноз пользователя выделен визуально.

### 7.5. Экран матча — состояние Resolved

- Аналогично Locked, дополнительно показывается:
  - Правильные ответы (отмечены иконками ✓/✗).
  - Начисленные очки за каждый вопрос (`WinnerPoints`, `VotedOutPoints`).
  - Общая сумма очков за матч (`TotalPoints`).

### 7.6. Таблица лидеров (LeaderboardTab)

- Ранжирование: Место | Аватар + Никнейм | Очки.
- Медали для Топ-3 (🥇🥈🥉).
- Строка текущего пользователя выделена визуально.
- Данные загружаются из blob-файла `leaderboard-{tournamentId}.json`.

### 7.7. Админ-элементы (inline в интерфейсе)

Функции администратора встроены в общий интерфейс (не отдельная страница):

- Кнопка **«+ Новый турнир»** на списке турниров.
- Кнопка **«+ Создать игру»** на странице турнира.
- **MatchStateControls** на каждой карточке матча:
  - `Upcoming`: [Открыть]
  - `Open`: [← Ожидание] [Заблокировать]
  - `Locked`: [← Открыть] [Завершить → ResolveForm]
  - `Resolved`: [← Отменить результат]
  - Кнопка удаления матча (на всех статусах, кроме Resolved).
- **ResolveForm** (модальное окно):
  - Переключатель «Победитель»: Мирные / Мафия.
  - Мультиселект слотов (1–10) или «Никто» (взаимоисключающие).
- **Статистика турнира**: число матчей по статусам, общее число прогнозов.

## 8. Обновление состояния в реальном времени

### 8.1. Механизм

Используется **polling Azure Blob Storage**, а не WebSockets/SSE:

1. Модератор меняет статус матча → Backend публикует агрегированное состояние в файл `match-state-{matchId}.json` в Azure Blob Storage.
2. Frontend периодически опрашивает blob-файлы и обновляет UI без перезагрузки.

### 8.2. Содержимое blob-файла (match-state)

```json
{
  "matchId": 1,
  "tournamentId": 1,
  "version": 1709123456789,
  "state": "Open",
  "updatedAt": "...",
  "tableSize": 10,
  "totalPredictions": 42,
  "winnerVotes": {
    "town": { "count": 30, "percent": 71.4 },
    "mafia": { "count": 12, "percent": 28.6 }
  },
  "votedOutVotes": [
    { "slot": 0, "count": 5, "percent": 11.9 }
  ],
  "matchResult": null
}
```

### 8.3. Адаптивные интервалы polling

| Контекст | Интервал |
|----------|----------|
| Матч `Open` или `Locked` | 2 сек |
| Матч `Resolved` | 15 сек |
| Матч `Upcoming` | 10–20 сек |
| Вкладка браузера неактивна | 20 сек |

Список матчей обновляется каждые 60 секунд. Первый `Open`-матч автоматически раскрывается со скроллом.

### 8.4. Серверный throttling публикации

Для матчей в статусе `Open` blob публикуется не чаще **1 раза в 10 секунд** (защита от flood при массовых прогнозах). Принудительная публикация доступна через админ-endpoint.

### 8.5. Кеширование

- `match-state-{id}.json`: `Cache-Control: max-age=1, must-revalidate`.
- `leaderboard-{id}.json`: `Cache-Control: max-age=5, must-revalidate`.

## 9. Корнер-кейсы и игровые правила

1. **Удаление за фолы:** Если игрок удален до голосования (4-й фол), голосование не проходит — правильный ответ для Вопроса №2: «Никто» (слот `0`).
2. **Множественный уход (попил):** Модератор выбирает несколько правильных слотов. Баллы начисляются по стандартной формуле каждой группе угадавших.
3. **АФК (пропуск прогноза):** Если пользователь не сделал прогноз до перехода в `Locked` — 0 баллов за этот матч.
4. **Откат результатов:** Модератор может вернуть матч из `Resolved` в `Locked`, при этом очки и результат удаляются, лидерборд пересчитывается.
5. **Переоткрытие:** Матч может быть возвращен из `Locked` в `Open` (пользователи снова могут менять прогнозы).

| **Бот** | Опционально (вне MVP): webhook/команды/уведомления Telegram |

### 10.1. Индексы БД

- `IX_PickemMatch_TournamentId_State` — быстрый поиск матчей по турниру и статусу.
- `IX_Prediction_MatchId` (INCLUDE: PredictedWinner, PredictedVotedOut) — покрывающий индекс для агрегации голосов.
- `IX_Prediction_UserId` — поиск прогнозов пользователя.
- `IX_Leaderboard_TournamentId_TotalPoints` — сортированный лидерборд.
- Unique constraints: `(MatchId, UserId)` на Prediction, `(TournamentId, UserId)` на Leaderboard, `MatchId` на MatchResult, `PredictionId` на PredictionScore.

## 12. Безопасность

1. **Telegram auth:** Валидация `initData` через HMAC-SHA256 (`secretKey = HMAC("WebAppData", botToken)`).
2. **JWT:** HS256-подпись, проверка issuer/audience/lifetime, ClockSkew = 0.
3. **Admin-доступ:** Allowlist `TelegramId` в конфигурации, проверяется при каждом запросе middleware.
4. **Публичные blob-файлы:** Содержат только агрегированную статистику, никаких персональных данных.
5. **Middleware pipeline:** `ExceptionHandlingMiddleware` → `TelegramAuthMiddleware` → function handler.
